# Протокол взаємодії з математичною моделлю по UDP

Цей документ описує детально взаємодію між емулятором ServoLib та математичною моделлю двигуна по протоколу UDP.

## Основна інформація

- **IP адреса сервера моделі**: 127.0.0.1 (локальний хост) або IP адреса комп'ютера з моделлю
- **Порт сервера моделі**: 8888
- **Порт клієнта емулятора**: 8889
- **Частота оновлення**: 1000 Гц (1 мс)
- **Таймаут**: 100 мс

## Структури даних

### UDP повідомлення
```c
typedef struct {
    UDP_MsgType_t msg_type;    // Тип повідомлення
    uint32_t sequence_number;  // Номер послідовності
    uint8_t payload[256];      // Тіло повідомлення
    uint16_t payload_size;     // Розмір тіла повідомлення
} UDP_Message_t;
```

### Типи повідомлень
```c
typedef enum {
    UDP_MSG_TYPE_MOTOR_CMD    = 0x01,  // Команда двигуну
    UDP_MSG_TYPE_SENSOR_CMD   = 0x02,  // Команда сенсору
    UDP_MSG_TYPE_BRAKE_CMD    = 0x03,  // Команда гальмам
    UDP_MSG_TYPE_MOTOR_STATE  = 0x10,  // Стан двигуна від моделі
    UDP_MSG_TYPE_SENSOR_STATE = 0x11,  // Стан сенсора від моделі
    UDP_MSG_TYPE_BRAKE_STATE  = 0x12,  // Стан гальм від моделі
    UDP_MSG_TYPE_CONFIG       = 0x20,  // Команда конфігурації
    UDP_MSG_TYPE_PING         = 0xFF   // Перевірка зв'язку
} UDP_MsgType_t;
```

## Взаємодія з моделлю двигуна

### 1. Встановлення з'єднання
1.1. Емулятор ініціалізує UDP сокет на порту 8889 (локально)
1.2. Емулятор встановлює адресу сервера моделі (IP:8888)
1.3. Відбувається перевірка з'єднання через команду PING
1.4. Якщо зв'язок встановлено, система переходить у режим нормальної роботи

### 2. Керування двигуном
2.1. Емулятор формує команду `UDP_MSG_TYPE_MOTOR_CMD`:
- Створює структуру `UDP_Motor_Cmd_t`:
  - `power`: значення потужності від -100.0 до +100.0
    - додатнє значення: пряме обертання
    - від'ємне значення: зворотне обертання
    - 0.0: зупинка
  - `target_position`: цільова позиція (використовується для деяких моделей)
  - `target_velocity`: цільова швидкість (використовується для деяких моделей)

2.2. Емулятор відправляє команду до моделі:
- Заповнює `UDP_Message_t`:
  - `msg_type` = `UDP_MSG_TYPE_MOTOR_CMD`
  - `payload` = структура `UDP_Motor_Cmd_t`
  - `payload_size` = розмір структури
- Викликає `sendto()` для відправки даних на IP:8888

2.3. Математична модель отримує команду та обробляє її:
- Виконує фізичне моделювання двигуна
- Розраховує нові параметри: позицію, швидкість, струм
- Визначає стан двигуна (заклинювання, перевищення струму)

2.4. Модель готує відповідь `UDP_MSG_TYPE_MOTOR_STATE`:
- Створює структуру `UDP_Motor_State_t`:
  - `position`: поточна позиція в градусах
  - `velocity`: поточна швидкість в град/с
  - `current`: поточний струм в mA
  - `stalled`: чи заклинений двигун (true/false)
  - `overcurrent`: чи перевищено струм (true/false)
  - `error_flags`: прапори помилок

2.5. Модель відправляє стан назад емулятору:
- Заповнює `UDP_Message_t`:
  - `msg_type` = `UDP_MSG_TYPE_MOTOR_STATE`
  - `payload` = структура `UDP_Motor_State_t`
  - `payload_size` = розмір структури
- Викликає `sendto()` для відправки даних на IP:8889

2.6. Емулятор отримує відповідь:
- Викликає `recvfrom()` з таймаутом
- Перевіряє `msg_type` на `UDP_MSG_TYPE_MOTOR_STATE`
- Розпаковує дані в структуру стану двигуна
- Оновлює внутрішній стан системи керування

## Взаємодія з системою гальм

### 1. Керування гальмами
1.1. Емулятор формує команду `UDP_MSG_TYPE_BRAKE_CMD`:
- Створює структуру `UDP_Brake_Cmd_t`:
  - `engaged`: true - активувати гальма, false - відпустити гальма
  - `timestamp`: час відправлення (для синхронізації)

1.2. Емулятор відправляє команду до моделі:
- Заповнює `UDP_Message_t`:
  - `msg_type` = `UDP_MSG_TYPE_BRAKE_CMD`
  - `payload` = структура `UDP_Brake_Cmd_t`
  - `payload_size` = розмір структури
- Викликає `sendto()` для відправки даних на IP:8888

1.3. Модель отримує команду та оновлює стан гальм:
- Застосовує фізичну модель гальм
- Враховує механіку гальмування до системи двигуна

1.4. Модель готує відповідь `UDP_MSG_TYPE_BRAKE_STATE`:
- Створює структуру `UDP_Brake_State_t`:
  - `engaged`: чи активні гальма (true/false)
  - `ready`: чи готові гальма до роботи (true/false)
  - `error_flags`: прапори помилок

1.5. Модель відправляє стан гальм:
- Заповнює `UDP_Message_t`:
  - `msg_type` = `UDP_MSG_TYPE_BRAKE_STATE`
  - `payload` = структура `UDP_Brake_State_t`
  - `payload_size` = розмір структури
- Викликає `sendto()` для відправки даних на IP:8889

1.6. Емулятор отримує стан гальм та оновлює внутрішній стан

## Взаємодія з сенсором

### 1. Отримання даних сенсора
1.1. Емулятор формує запит `UDP_MSG_TYPE_SENSOR_CMD`:
- В більшості випадків це просто запит на отримання даних
- Може містити команду підтвердження готовності

1.2. Емулятор відправляє запит до моделі:
- Заповнює `UDP_Message_t`:
  - `msg_type` = `UDP_MSG_TYPE_SENSOR_CMD`
  - `payload` = порожній або підтвердження
  - `payload_size` = 0 або розмір команди
- Викликає `sendto()` для відправки даних на IP:8888

1.3. Модель генерує відповідь на основі моделі двигуна:
- Використовує модель розташування сенсора відносно валу двигуна
- Враховує позицію двигуна для обчислення кута сенсора
- Може додавати шум до сигналу для реалістичності

1.4. Модель готує відповідь `UDP_MSG_TYPE_SENSOR_STATE`:
- Створює структуру `UDP_Sensor_State_t`:
  - `angle`: кут у градусах
  - `velocity`: швидкість у град/с
  - `connected`: чи сенсор підключений (true/false)
  - `error_flags`: прапори помилок

1.5. Модель відправляє дані сенсора:
- Заповнює `UDP_Message_t`:
  - `msg_type` = `UDP_MSG_TYPE_SENSOR_STATE`
  - `payload` = структура `UDP_Sensor_State_t`
  - `payload_size` = розмір структури
- Викликає `sendto()` для відправки даних на IP:8889

1.6. Емулятор отримує дані сенсора та використовує їх для керування

## Синхронізація та таймінги

### Цикл оновлення (1000 Гц)
1. Емулятор відправляє всі команди (двигун, гальма, сенсор)
2. Емулятор чекає відповіді з таймаутом 100 мс
3. Емулятор обробляє всі отримані стани
4. Емулятор оновлює внутрішній стан системи керування
5. Очікування до наступного циклу (1 мс)

### Обробка помилок
- Якщо відповідь не отримано протягом таймауту - використовується попередній стан
- Якщо кількість помилок перевищує поріг - система переходить у аварійний режим
- Помилки логуються для діагностики

## Оптимізація комунікації

### Буферизація
- Команди можуть бути буферизовані перед відправкою
- Стан може бути буферизований перед обробкою
- Використовується пакетна передача даних для ефективності

### Стиснення даних
- Використовуються компактні структури даних
- Мінімізується розмір UDP пакетів
- Використовується ефективне серіалізовання

Цей протокол забезпечує ефективну реал-тайм взаємодію між емулятором ServoLib та математичною моделлю для точного тестування алгоритмів керування.
