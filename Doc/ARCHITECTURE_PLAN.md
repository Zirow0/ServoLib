# План архітектурних рішень ServoLib

**Версія:** 2.0
**Дата:** 2025-12-29
**Статус:** Планування рефакторингу

---

## Огляд

Цей документ описує плановані архітектурні зміни в бібліотеці ServoLib для покращення модульності, спрощення інтерфейсів та підтримки різних типів моторів (DC, BLDC, Stepper).

---

## Основні принципи

### 1. Розділення відповідальності

**Motor Interface/Driver:**
- Відповідає **тільки** за керування мотором
- Виконує команди без перевірок безпеки
- Не знає про струм, температуру, обмеження швидкості
- Простий виконавець: "встановити потужність" → виконує

**Safety System:**
- Окремий модуль для всіх перевірок безпеки
- Моніторинг: струм, температура, напруга, положення, швидкість
- Визначає сценарії реакції на помилки
- Не керує мотором напряму

**Servo Controller:**
- Центральний координатор системи
- Отримує дані від датчиків
- Викликає Safety для перевірок
- При ініціалізації зв'язує модулі через вказівники на функції
- Відправляє команди на Motor Driver

**FOC Module:**
- Універсальний модуль керування
- Отримує вказівник на інтерфейс двигуна
- Автоматично визначає режим роботи за типом мотора
- Для DC мотора перенаправляє виклики напряму на Motor Driver

### 2. Потік даних

```
Датчики (позиція, струм, температура, напруга)
    ↓
Servo Controller (координація)
    ↓
Safety System (перевірка безпеки) → OK/ERROR + сценарій
    ↓
FOC Module / PID каскад (обчислення керування)
    ↓
Motor Driver (виконання команд)
```

## Інтерфейс мотора

### Концепція універсального інтерфейсу

**Проблема:** Різні типи моторів потребують різних команд керування
- DC мотор: одна потужність (-100% до +100%)
- Кроковий мотор: дві фази (phase_a, phase_b)
- BLDC мотор: три фази (phase_a, phase_b, phase_c)

**Рішення:** Універсальна команда з union

Команда керування містить:
- Тип мотора (enum: DC / Stepper / BLDC)
- Union з даними для кожного типу
- Драйвер перевіряє відповідність типу команди

### Спрощення структур

**Видаляється з Motor_Params_t:**
- `max_current` - переноситься в Safety
- `max_temperature` - переноситься в Safety
- `max_rpm` - переноситься в Safety

**Залишається:**
- `type` - тип мотора
- `max_power` - максимальна потужність
- `min_power` - мінімальна потужність
- `invert_direction` - інверсія напрямку

**Видаляється з Motor_Stats_t:**
- `current_draw` - вимірюється окремим датчиком

### Wrapper функції

Для зручності використання створюються inline функції:
- `Motor_SetPower_DC(motor, power)` - для DC моторів
- `Motor_SetPower_Stepper(motor, phase_a, phase_b)` - для крокових
- `Motor_SetPower_BLDC(motor, phase_a, phase_b, phase_c)` - для BLDC

## Система безпеки

### Інтеграція з Servo Controller

**Архітектурне рішення:** Safety як окремий модуль, викликається з Servo Controller

**Обґрунтування:**
- Простота синхронізації - все в одному control loop
- Швидка реакція - гарантований порядок виконання
- Надійність - чіткий control flow
- Embedded-friendly - один цикл на фіксованій частоті

### Відповідальність Safety

**Моніторинг параметрів:**
- Положення (min/max межі)
- Швидкість (максимальна швидкість)
- Прискорення (максимальне прискорення)
- Струм (максимальний струм, таймаут перевантаження)
- Температура (максимальна температура)
- Напруга (мінімальна/максимальна напруга живлення)
- Watchdog (таймаут оновлення)

**Джерела даних:**
- Датчик позиції (encoder)
- Датчик струму
- Датчик температури
- Датчик напруги
- Обчислена швидкість (з позиції)

**Реакція на порушення:**
- Встановлення прапорців порушень
- Логування помилок через Error Manager
- Повернення статусу ERROR
- Servo Controller виконує emergency stop

### Гнучкість конфігурації

Кожен тип захисту може бути увімкнений/вимкнений:
- DC мотор: тільки позиція, швидкість, watchdog
- BLDC мотор: + струм, температура, напруга
- Прецизійний позиціонер: + прискорення

## Каскадне PID регулювання

### Трьохступеневий контур керування

**Рівень 1: Контур моменту (внутрішній, найшвидший)**
- Вхід: цільовий момент
- Зворотній зв'язок: струм (струм пропорційний моменту)
- Вихід: напруга/потужність на мотор
- Частота: найвища (10+ kHz для BLDC)

**Рівень 2: Контур швидкості (середній)**
- Вхід: цільова швидкість
- Зворотній зв'язок: швидкість (розраховується в драйвері положення)
- Вихід: цільовий момент для контуру моменту
- Частота: середня (1-10 kHz)

**Рівень 3: Контур позиції (зовнішній, найповільніший)**
- Вхід: цільова позиція
- Зворотній зв'язок: позиція (з драйвера положення)
- Вихід: цільова швидкість для контуру швидкості
- Частота: базова (1 kHz)

### Переваги каскадного регулювання

- Кожен контур оптимізований для своєї задачі
- Швидка реакція на збурення (внутрішній контур)
- Точне позиціонування (зовнішній контур)
- Обмеження струму/моменту на найнижчому рівні

## FOC Module

### Призначення

Універсальний модуль керування, який:
- Отримує вказівник на інтерфейс двигуна при ініціалізації
- Автоматично визначає режим роботи за типом мотора
- Інкапсулює специфіку керування різними типами моторів

### Режими роботи

**DC мотор:**
- FOC Module перенаправляє виклики напряму на Motor Driver
- Використовується простий PID для позиції
- Контур моменту не використовується (або спрощений)

**Кроковий мотор (2 фази):**
- FOC Module генерує сигнали для двох фаз
- Підтримка мікрокроків
- Синусоїдальна комутація

**BLDC мотор (3 фази):**
- Повний FOC алгоритм
- Clarke/Park перетворення
- SVPWM модуляція
- Контур струму (id, iq)

### Автоматичний вибір режиму

При ініціалізації FOC Module:
1. Отримує вказівник на Motor Interface
2. Читає тип мотора з параметрів
3. Зв'язує внутрішні функції з відповідними алгоритмами
4. Подальші виклики йдуть через вказівники (без switch/case в runtime)

---

## Підтримка різних типів моторів

### DC мотори

**Драйвер:** `pwm.c`, `pwm_udp.c`

**Характеристики:**
- Одноканальне керування (потужність -100% до +100%)
- Двоканальний режим для H-bridge (forward/reverse)
- Підтримка інверсії напрямку
- PWM частота конфігурується

**Регулювання:**
- Спрощений каскад: позиція → швидкість → потужність
- Контур моменту опціональний (якщо є датчик струму)

### BLDC мотори

**Модуль:** FOC Module

**Характеристики:**
- Триканальне керування (3 фази)
- FOC (Field-Oriented Control) режим
- Потребує датчика кута ротора
- Контроль струму по фазах

**Регулювання:**
- Повний 3-ступеневий каскад
- Контур моменту обов'язковий

**Функції FOC:**
- Clarke transform (3-phase → α,β)
- Park transform (α,β → d,q)
- PI регулятори струму
- Inverse Park (d,q → α,β)
- SVPWM модуляція

### Крокові мотори

**Модуль:** FOC Module (2-фазний режим)

**Характеристики:**
- Двоканальне керування (phase_a, phase_b)
- Підтримка мікрокроків
- Синусоїдальна комутація

**Регулювання:**
- Каскад: позиція → швидкість → фази
- Контур моменту через струм фаз

## Датчики

### Принцип роботи

Датчики є незалежними модулями:
- Надають дані Servo Controller
- Не приймають рішень
- Не викликають Safety напряму

### Типи датчиків

**Датчики позиції:**
- AS5600 (12-bit I2C) - для DC моторів
- AEAT-9922 (18-bit SPI) - для BLDC і прецизійних систем
- UDP емуляція

**Датчики струму:**
- АЦП з шунтом
- Інтегровані в драйвери BLDC
- Для контуру моменту

**Датчики температури:**
- NTC термістори
- Вбудовані в драйвери
- Для Safety системи

**Датчики напруги:**
- Дільник напруги + АЦП
- Моніторинг напруги живлення
- Захист від перенапруги/недонапруги

### Інтеграція з Safety

Servo Controller:
1. Читає всі датчики
2. Передає дані в Safety_Update()
3. Отримує результат перевірки
4. Реагує відповідно

---

## Servo Controller

### Роль в системі

Центральний модуль, який:
- Ініціалізує всі компоненти
- Координує роботу в control loop
- Виконує послідовність операцій щокадру

### Ініціалізація (одноразово)

При ініціалізації Servo Controller:
1. Отримує тип мотора з конфігурації
2. Зв'язує FOC Module з Motor Interface
3. Налаштовує каскадні PID контролери
4. Всі зв'язки через вказівники на функції
5. В runtime немає switch/case - тільки виклики через вказівники

### Цикл оновлення

Послідовність операцій:
1. Читання всіх датчиків (позиція, струм, температура, напруга)
2. Виклик Safety_Update() для перевірки
3. Якщо безпечно - виклик каскадного регулятора через вказівники
4. Якщо небезпечно - виклик emergency stop

---

## Конфігурація

### Система конфігурації залишається незмінною

Три рівні:
1. `config_lib.h` - константи бібліотеки
2. `config_user.h` - параметри проекту (найвищий пріоритет)
3. `config_defaults.h` - значення за замовчуванням

### Safety параметри в config_user.h

Користувач налаштовує:
- `MOTOR_MAX_CURRENT` - для свого мотора
- `MOTOR_MAX_TEMPERATURE` - для свого обладнання
- `MOTOR_MIN_VOLTAGE`, `MOTOR_MAX_VOLTAGE` - межі напруги живлення
- `DEFAULT_MAX_VELOCITY` - для свого застосування
- Увімкнення/вимкнення окремих захистів

---

## Ризики та обмеження

**Складність FOC Module:**
- Інкапсулює багато логіки
- Потребує якісного тестування

**Вимоги до датчиків для BLDC:**
- Обов'язковий датчик струму для контуру моменту
- Високороздільний енкодер для FOC

**Частота оновлення:**
- Контур моменту потребує високої частоти (10+ kHz)
- Може бути обмеженням для деяких MCU

---

## Висновок

Планована архітектура забезпечує:
- Чітке розділення відповідальності між модулями
- Універсальний інтерфейс для різних типів моторів
- 3-ступеневе каскадне PID регулювання (момент → швидкість → позиція)
- FOC Module як універсальний контролер з автоматичним вибором режиму
- Централізовану систему безпеки з моніторингом напруги
- Ефективну роботу через вказівники на функції (без switch/case в runtime)
